The Problem

Libraries are blind to themselves.

A student leaving their hostel room to study has no way of knowing whether the library is too crowded, too noisy, too stuffy, or simply too hot — until they arrive and it's already too late.

Library administrators, on the other hand, have no instrumented data on their space. Decisions about air conditioning, seating, and ventilation are made on anecdote, not evidence.

Commercial solutions (Nest, Arlo, smart building platforms) cost upward of ₹10,000 and solve access control — not environment quality. The gap this project fills is environmental intelligence for spaces that can't justify enterprise infrastructure.

The library's condition changes every hour. Its awareness of itself doesn't change at all.

Pipeline

How it works

01
🔬
Sense
Six sensors continuously measure temperature, humidity, sound level, air quality, proximity, and motion — every 2 seconds without blocking the main loop.
02
🧠
Process
The ESP32 runs directional occupancy logic on dual ultrasonics, applies temperature correction to distance math, and computes a single Comfort Score from all six inputs.
03
☁️
Broadcast
Nine JSON keys are pushed to Firebase Realtime Database every 5 seconds. The custom Mobile PWA receives live updates via a persistent WebSocket stream.

Hardware

Every component has a reason.

Sensor · Input
DHT22
GPIO 4 · Single-Wire
Measures temperature and humidity simultaneously — two parameters from one sensor. Temperature reading also corrects the ultrasonic speed-of-sound calculation.
🌡
Sensor · Input
HC-SR04 ×2
GPIO 26/27 + 32/33 · TRIG/ECHO
Two ultrasonics placed at the entrance, offset by ~20cm. Direction of travel is inferred from which one triggers first — enabling a bidirectional people counter without RFID or cameras.
📡
Sensor · Input
PIR HC-SR501
GPIO 14 · Digital
Passive infrared detects occupant presence. Combined with the ultrasonic count, it confirms whether the space is actively used or simply has residual count from a previous session.
👁
Sensor · Analog
Microphone KY-037
GPIO 34 · 12-bit ADC
Ambient sound level as an analog voltage proxy for decibels. This is the most differentiated metric — no commercial people counter reports whether it's actually quiet enough to study.
🔊
Sensor · Analog
MQ-2 Gas
GPIO 35 · 12-bit ADC
Detects CO, LPG, smoke, and general combustible gases. Used here as an air quality proxy — crowded enclosed rooms generate CO₂ and VOC accumulation that the MQ-2 partially captures.
🌿
Actuators · Output ×3
LED + Buzzer
GPIO 18, 19, 23 · Digital
Green LED = conditions acceptable. Red LED = capacity reached or poor air. Buzzer = gas threshold breach or crowd + motion event. All three also controllable remotely via the custom PWA Dashboard.

The Algorithm

Six readings.
One number.

Raw sensor data is noisy, context-dependent, and hard to act on. The Comfort Score distills everything into a single value between 0 and 100 — so a student can decide in one glance whether to walk over.

Each parameter carries a penalty weight proportional to its impact on study quality. Air quality carries the heaviest single penalty — poor air drives fatigue faster than any other factor.

C = max(0, 100 − 3×max(0, T − 30) // temp penalty − 2×max(0, H − 70) // humidity − 20 × [noise > 2500] // noise step − 25 × [air > 2000] // air quality − 30 × [count ≥ maxCap] // crowding )

What Makes It Different

Beyond the tutorial.

01
🎯
Directional Counting
Standard tutorials use one ultrasonic to detect presence. We use two, offset at the entrance, to determine direction of travel from trigger sequence — a true bidirectional people counter at zero extra chip cost.
02
🌡
Temperature-Corrected Distance
Speed of sound varies with air temperature. Using the DHT22 reading to compute v = 331.3 + 0.606×T eliminates a 3.5% systematic error that would cause missed entry detections at room temperature extremes.
03
📊
Computed Comfort Metric
Rather than displaying raw ADC values, the system computes a single human-readable score. This is the output that matters — actionable at a glance without requiring the user to interpret sensor physics.
04
⏱
Non-Blocking Architecture
Firebase Realtime Stream listeners handle all remote commands. No delay() anywhere in the firmware. The loop() remains perpetually responsive to cloud commands — critical for real-time remote actuator control.

Code Highlight

readDistance() — temp corrected 
float readDistance(int trig, int echo) {
  // DHT22 corrects speed of sound
  float v = 331.3 + (0.606 * temperature);

  digitalWrite(trig, LOW);
  delayMicroseconds(2);
  digitalWrite(trig, HIGH);
  delayMicroseconds(10);
  digitalWrite(trig, LOW);

  long dur = pulseIn(echo, HIGH, 30000);
  if (dur == 0) return 999.0;
  return (dur * v / 10000.0) / 2.0;
}
computeComfortScore() 
int computeComfortScore() {
  int score = 100;

  if (temperature > TEMP_MAX)
    score -= (int)((temperature-TEMP_MAX)*3);
  if (humidity > HUMIDITY_MAX)
    score -= (int)((humidity-HUMIDITY_MAX)*2);
  if (noiseLevel > NOISE_THRESHOLD)
    score -= 20;
  if (airQuality > GAS_THRESHOLD)
    score -= 25;
  if (occupancyCount >= maxCapacity)
    score -= 30;

  return constrain(score, 0, 100);
}
